# Интеграция ВК-бота «Альбатрос» с Битрикс24

Бот работает в двух режимах. По умолчанию (`DATA_PROVIDER=local`) он использует
локальную SQLite-базу и JSON-файл — это позволяет запускать его без интернета,
быстро тестировать сценарии и проводить демонстрации. После настройки Битрикс24
переменная переключается в `bitrix`, и бот начинает читать абонементы, тренеров
и расписание из CRM. Само по себе это переключение не требует изменения кода
бота — меняется только содержимое `.env`.

В этой инструкции описано, что нужно сделать в Битрикс24 один раз, чтобы бот
мог работать в режиме CRM.

---

## 1. Создание входящего вебхука

1. Войдите в Битрикс24 под учётной записью администратора портала.
2. Откройте раздел **Разработчикам → Другое → Входящий вебхук**.
3. Нажмите **Добавить вебхук**.
4. В списке прав отметьте:
   - **CRM (crm)** — нужно для чтения контактов и сделок;
   - **Списки (lists)** — нужно для чтения тренеров и расписания.
5. Сохраните вебхук. Битрикс выдаст URL вида:

   ```
   https://albatros.bitrix24.ru/rest/1/abc123def456ghi789/
   ```

6. Скопируйте этот URL в переменную `BITRIX_WEBHOOK_URL` в файле `.env`
   рядом с ботом.

> ⚠️ Этот URL равноценен паролю — кто его получит, тот сможет читать ваши
> контакты и сделки. Не публикуйте его в открытых репозиториях и не пересылайте
> в мессенджерах.

---

## 2. Поле «Номер карты» у Контакта

Каждый клиент клуба заводится в Битриксе как **Контакт**. Чтобы бот мог
найти абонемент по номеру карты, к карточке Контакта добавляется одно
дополнительное поле.

1. Откройте раздел **CRM → Контакты → Настройки → Настройка пользовательских полей**.
2. Нажмите **Добавить поле**, выберите тип **Строка**.
3. Параметры:
   - Заголовок (русский): **Номер карты «Альбатрос»**;
   - Код поля: оставьте автоматический (Битрикс назначит `UF_CRM_CARD_ID` или похожий);
   - Множественное: нет;
   - Обязательное: рекомендуется поставить «да».
4. Сохраните.

После этого в карточке каждого Контакта появится поле «Номер карты «Альбатрос»»,
куда администратор записывает номер вида `ALB-001`.

---

## 3. Воронка «Абонементы» (Сделки)

Один клиент может купить несколько абонементов за свою историю — поэтому
сами абонементы хранятся не в Контакте, а в **Сделках**.

1. **CRM → Сделки → Настройки → Воронки**.
2. Создайте воронку **Абонементы**.
3. В пользовательских полях Сделки добавьте два поля:
   - **Тип:** Число; **Заголовок:** *Осталось занятий*; код — `UF_CRM_SESSIONS_LEFT`.
   - **Тип:** Дата; **Заголовок:** *Действителен до*; код — `UF_CRM_VALID_UNTIL`.

При продаже абонемента администратор:

1. Открывает Контакт (или создаёт нового).
2. Заполняет «Номер карты «Альбатрос»».
3. Создаёт связанную Сделку из воронки «Абонементы», заполняет «Осталось занятий»
   и «Действителен до».

Бот, получив от клиента номер карты, найдёт этот Контакт, его последнюю
открытую Сделку из воронки «Абонементы» и покажет её клиенту.

---

## 4. Универсальный список «Тренеры»

1. **Сайты → Универсальные списки → Создать новый список**.
2. Название: **Тренеры**.
3. После создания зайдите в **Настройка полей** и добавьте:
   - Поле **Имя элемента (NAME)** уже существует — здесь будет ФИО тренера.
   - **Специализация** (тип «Список» или «Строка»; код пусть будет `SPEC`).
   - **Описание** (тип «Текст» или «HTML/текст»; код `DESC`).

Запишите идентификатор созданного списка — он понадобится в `.env`. Идентификатор
виден в адресной строке при открытии списка: `lists/?livefeed_listid=42` →
`BITRIX_TRAINERS_IBLOCK_ID=42`.

После этого карточки тренеров заполняются администратором через стандартный
веб-интерфейс Битрикса. Бот будет показывать их клиентам с задержкой
обновления до 60 секунд (внутренний кеш).

---

## 5. Универсальный список «Расписание»

Создаётся аналогично «Тренерам». Поля:

| Поле                       | Тип                            | Код             |
| -------------------------- | ------------------------------ | --------------- |
| День недели                | Список («Пн», «Вт» … «Вс»)     | `DAY`           |
| Время                      | Строка («09:00», «18:30»)      | `TIME`          |
| Направление                | Список (Йога, Пилатес, …)      | `DIRECTION`     |
| Тренер                     | Строка или привязка к спр.     | `TRAINER`       |
| Зал                        | Строка                         | `HALL`          |

Идентификатор списка укажите в `.env` как `BITRIX_SCHEDULE_IBLOCK_ID`.

---

## 6. Заполнение `.env`

Переключение в режим Битрикса:

```env
# Текущие переменные (остаются)
VK_TOKEN=...
GROUP_ID=...
OPERATOR_CHAT_ID=...

# Новые переменные
DATA_PROVIDER=bitrix
BITRIX_WEBHOOK_URL=https://albatros.bitrix24.ru/rest/1/abc123def456ghi789/
BITRIX_TRAINERS_IBLOCK_ID=42
BITRIX_SCHEDULE_IBLOCK_ID=43
```

Перезапустите бот (`python main.py`). В логе появится строка
`[PROVIDER] BitrixProvider запущен; webhook=…`. Это значит, что бот теперь
читает данные из CRM.

---

## 7. Как переключиться обратно в локальный режим

В `.env` поменяйте одну строку:

```env
DATA_PROVIDER=local
```

Перезапустите бот. Он снова начнёт работать с SQLite и `data/data.json`.
Это удобно для тестирования и на случай, если CRM временно недоступен.

---

## 8. Что делать, если бот не находит абонемент в режиме Битрикса

Стандартный порядок проверки:

1. В карточке Контакта поле «Номер карты «Альбатрос»» заполнено? Совпадает ли регистр?
   (Бот приводит ввод клиента к верхнему регистру — поле в Битриксе тоже должно
   быть в верхнем.)
2. Связана ли с этим Контактом хотя бы одна **открытая** Сделка из воронки
   «Абонементы»? Закрытые сделки бот игнорирует.
3. Заполнены ли в Сделке поля «Осталось занятий» и «Действителен до»?
4. Логи бота: при ошибке API Битрикс пишет понятное сообщение, например
   `[BITRIX] API-ошибка crm.contact.list: QUERY_LIMIT_EXCEEDED`.
5. Лимит вебхука — 2 запроса в секунду. На реальной нагрузке клуба этого
   с большим запасом, но при массовом тестировании может быть превышен.

Если ничего не помогло — переключите бота в локальный режим (`DATA_PROVIDER=local`),
чтобы клиенты продолжали получать ответы, и разбирайтесь с CRM спокойно.

---

## 9. Что бот **не** делает (пока)

- Не создаёт сделки сам — это всегда ручная работа администратора в Битриксе.
- Не отмечает посещения (декремент `UF_CRM_SESSIONS_LEFT`) — пока ведёт администратор.
- Не уведомляет клиентов об окончании срока — это запланировано в перспективе
  (см. подраздел 7.2 ВКР).

Эти возможности добавляются на следующих этапах, когда у клуба появится
устойчивая практика работы с CRM.
